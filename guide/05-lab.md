# 5. 실습

<div class="badge-row"><span>난이도: 입문</span><span>권장 시간: 25~40분</span><span>대상: labs/task-board</span></div>

## 실습 목표

실습 프로젝트에는 의도적으로 버그와 미구현 요구사항이 포함되어 있습니다.

<ul class="checklist">
<li>코드를 먼저 이해한다.</li>
<li>실패 테스트를 재현한다.</li>
<li>원인과 계획을 먼저 받는다.</li>
<li>최소 범위로 수정한다.</li>
<li>테스트와 diff로 검증한다.</li>
</ul>

## 준비

저장소 루트에서 실행합니다.

```bash
cd labs/task-board
npm test
claude
```

## 실습 1 — 프로젝트 탐색

<div class="lab-box">
<h3>목표: 수정 없이 코드 구조 파악</h3>

```text
이 프로젝트의 목적, 파일별 역할, 데이터 흐름을 설명해줘.
아직 파일을 수정하거나 명령을 실행하지 마.
설명의 근거가 되는 파일과 함수명을 함께 제시해줘.
```
</div>

확인 포인트:

- `TaskBoard`의 주요 메서드
- 테스트가 기대하는 동작
- 입력 검증 위치

## 실습 2 — 실패 테스트 분석

<div class="lab-box">
<h3>목표: 증상과 근본 원인 구분</h3>

```text
npm test를 실행해 실패를 재현해줘.
아직 수정하지 말고 각 실패의 재현 조건, 근본 원인, 최소 수정안을 정리해줘.
```
</div>

발표자가 질문할 내용:

1. 테스트 실패 메시지만 고치면 되는가?
2. 같은 원인으로 발생할 수 있는 다른 엣지 케이스는 무엇인가?
3. 수정 범위를 어디까지 제한해야 하는가?

## 실습 3 — 버그 수정

<div class="lab-box">
<h3>목표: 최소 수정 + 회귀 방지</h3>

```text
앞서 확인한 원인만 최소 범위로 수정해줘.
공백 제목이 거부되는 테스트를 추가하고 전체 테스트를 실행해줘.
관련 없는 리팩터링은 하지 마.
```
</div>

완료 기준:

- 빈 문자열과 공백 제목을 거부
- ID가 삭제 이후에도 중복되지 않음
- 전체 테스트 통과

## 실습 4 — 기능 추가

<div class="lab-box">

```text
수정 전 계획을 먼저 보여줘.
```
</div>

## 실습 5 — 변경 리뷰

```text
현재 git diff를 리뷰해줘.
정확성, 회귀 가능성, 테스트 누락을 우선순위 순으로 정리해줘.
문제가 없으면 검증한 근거와 남은 위험을 알려줘.
```

## 선택 실습 — 프로젝트 지침 만들기

```text
이 프로젝트를 분석해 간결한 CLAUDE.md 초안을 만들어줘.
실제 존재하는 명령만 기록하고, 테스트 우선·최소 변경·검증 규칙을 포함해줘.
```

::: details 진행자용 정답 방향
- `addTask()`는 `title.trim()` 결과를 검증해야 합니다.
- `tasks.length + 1` 방식은 삭제 후 ID 중복을 만들 수 있으므로 별도 증가 카운터가 필요합니다.
- `getPendingTasks()`는 `completed === false`인 항목만 반환해야 합니다.
- 반환 배열을 외부에서 수정해 내부 상태가 변하지 않도록 복사 여부를 토론할 수 있습니다.
:::
