# 4. 잘 시키는 방법

<div class="badge-row"><span>난이도: 입문</span><span>권장 시간: 10분</span><span>핵심: 목적·범위·제약·검증</span></div>

## 좋은 요청의 5요소

1. **목적** — 무엇을 해결할지
2. **범위** — 어디까지 다룰지
3. **제약** — 유지해야 할 것과 금지사항
4. **절차** — 먼저 분석할지, 계획을 받을지
5. **완료 기준** — 어떤 테스트가 통과해야 하는지

## 범용 템플릿

```text
[목적]
사용자가 빈 제목으로 할 일을 추가할 수 있는 버그를 수정해줘.

[절차]
먼저 관련 코드와 테스트를 찾아 원인을 설명해줘.
수정 전에 변경 계획을 제시해줘.

[범위·제약]
외부 API와 저장 데이터 형식은 변경하지 마.
관련 없는 리팩터링은 하지 마.

[완료 기준]
빈 제목과 공백 제목이 모두 거부되어야 하고,
기존 테스트와 새 테스트가 모두 통과해야 해.
```

## 작업별 요청 예시

### 코드 이해

```text
인증 요청이 들어와 권한 검사가 끝날 때까지의 흐름을 설명해줘.
관련 파일과 함수명을 순서대로 제시하고, 확인되지 않은 부분은 추정이라고 표시해줘.
```

### 버그 수정

```text
실패 테스트를 먼저 실행하고 원인을 확인해줘.
아직 수정하지 말고, 재현 조건·근본 원인·최소 수정안을 정리해줘.
```

### 기능 구현

```text
기존 패턴과 테스트 스타일을 먼저 확인해줘.
구현 계획을 제시한 뒤 승인된 범위만 수정하고 테스트까지 실행해줘.
```

### 리팩터링

```text
동작 변경 없이 중복만 제거해줘.
리팩터링 전후에 동일한 테스트를 실행하고, public API 변화가 없는지 확인해줘.
```

### 리뷰

```text
현재 diff만 리뷰해줘.
정확성, 회귀 가능성, 보안, 테스트 누락 순으로 중요도와 근거를 제시해줘.
스타일 취향은 제외해줘.
```

## 실패하기 쉬운 요청과 개선

| 나쁜 요청 | 문제 | 개선 |
|---|---|---|
| `이 코드 개선해줘` | 범위·기준 없음 | 성능/가독성/버그 중 목표 지정 |
| `알아서 다 해줘` | 과도한 변경 | 분석→계획→최소 수정 단계화 |
| `테스트도 해줘` | 어떤 테스트인지 모호 | 명령과 완료 기준 명시 |
| `이 방식으로 구현해` | 기존 구조 무시 가능 | 기존 패턴 먼저 조사 요청 |

## 가장 중요한 습관

::: tip 한 단계씩 검증하기
**큰 요청 하나보다, 분석 → 계획 → 구현 → 테스트 → 리뷰의 작은 요청 다섯 개가 더 안전하고 정확합니다.**
:::
