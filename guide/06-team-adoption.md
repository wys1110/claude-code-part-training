# 6. 팀 적용 방법

<div class="badge-row"><span>난이도: 중급</span><span>권장 시간: 5~10분</span><span>핵심: 개인 생산성보다 팀 재현성</span></div>

## 권장 도입 순서

### 1단계 — 안전한 개인 활용

- 공개 예제 또는 비민감 코드에서 연습
- 코드 탐색, 테스트 설명, 문서 초안부터 시작
- 변경은 작은 단위로 제한

### 2단계 — 팀 규칙 정리

프로젝트 루트 `CLAUDE.md`에 아래만 먼저 기록합니다.

- 설치·빌드·테스트·린트 명령
- 핵심 아키텍처와 금지된 의존 방향
- 코드 스타일보다 중요한 품질 기준
- 완료 전 반드시 실행할 검증

### 3단계 — 반복 업무를 Skill로 만들기

후보 선정 기준:

- 매주 반복된다
- 담당자마다 결과 품질 편차가 크다
- 체크리스트가 이미 존재한다
- 입력과 산출물 형식이 비교적 명확하다

### 4단계 — 자동화와 외부 연결

- Hooks: formatter, lint, 정책 검사
- MCP: 업무 시스템 조회와 제한된 작업
- CI: 리뷰·문서·테스트 보조

## 추천 파일 구조

```text
project/
├─ CLAUDE.md
└─ .claude/
   ├─ settings.json
   ├─ skills/
   │  └─ review-pr/
   │     └─ SKILL.md
   ├─ agents/
   └─ hooks/
```

## 팀 운영 원칙

| 원칙 | 적용 방식 |
|---|---|
| 최소 권한 | 읽기부터 시작하고 쓰기·실행 권한은 필요한 범위만 허용 |
| 재현 가능 | 사용한 명령과 결과를 남기고 CI로 다시 확인 |
| 작은 변경 | 한 작업·한 목적 중심의 diff 유지 |
| 사람의 승인 | merge, 배포, 데이터 변경은 담당자가 최종 판단 |
| 지속 개선 | 실패 사례를 `CLAUDE.md` 또는 Skill에 반영 |

## 파일럿 과제 추천

1. 오래된 모듈 구조 설명서 생성
2. 실패 테스트 원인 분석
3. 테스트 없는 유틸 함수 테스트 추가
4. PR 변경점 요약과 리뷰 체크리스트
5. 반복되는 문서 업데이트

::: tip 성공 지표
코드 생성량보다 **탐색 시간 감소, 리뷰 누락 감소, 테스트 실행률, 반복 업무 리드타임**을 측정하는 것이 좋습니다.
:::
