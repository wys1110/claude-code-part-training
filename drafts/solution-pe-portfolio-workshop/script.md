# Claude Code로 업무 포트폴리오 만들기 — 발표자 스크립트

## Slide 1 — Claude Code로 업무 포트폴리오 만들기

[약 1분]

오늘은 Solution PE팀 Staff가 Claude Code를 설명으로만 듣는 시간이 아니라, 빈 저장소에서 개인 업무 포트폴리오를 만들고 공개 URL까지 확인하는 과정을 함께 밟는 워크숍입니다. 발표자는 원용석이며, 개발 경험보다 작업을 안전하게 맡기고 검증하는 방법에 초점을 맞춘다고 안내합니다.

먼저 두 시간 뒤 각자 손에 남아야 할 결과부터 보겠습니다.

## Slide 2 — 오늘은 공개 URL 하나를 완성한다

[약 1분]

오늘의 산출물은 문서나 아이디어 목록이 아니라 실제로 열리는 공개 URL 하나입니다. 빈 저장소, 포트폴리오 파일, 확인된 공개 주소라는 세 지점을 손으로 연결해 봅니다. 중간 도구보다 이 끝점이 중요하다고 강조합니다.

그렇다면 무엇을 보고 “완성됐다”고 판단할지 다음 장에서 정하겠습니다.

## Slide 3 — 완료는 메시지가 아니라 세 가지 증거로 판단한다

[약 1분]

Claude Code가 “완료했습니다”라고 말해도 그 문장 자체는 증거가 아닙니다. 화면이 읽히는지, Git의 변경이 의도와 같은지, 공개 URL이 실제로 열리는지를 각각 확인해야 합니다. 참가자에게 세 증거 중 평소 가장 자주 놓치는 것이 무엇인지 손으로 표시하게 합니다.

증거를 모으기 전에, 공개 작업에서 지켜야 할 안전 경계를 먼저 세우겠습니다.

## Slide 4 — 공개 가능한 정보만 사용하고 중요한 행동은 확인한다

[약 2분]

왼쪽의 공개 가능 항목과 공개 금지 항목을 소리 내어 구분합니다. 역할이나 일반화한 업무 방식은 쓸 수 있지만 고객명, 내부 수치, 사내 URL, 토큰은 넣지 않습니다. 삭제·Commit·Push는 결과를 되돌리거나 외부에 영향을 주므로 사람이 확인한 뒤 실행한다고 못 박습니다.

참가자에게 자신의 소개 문장에 실제 회사 자료가 섞일 가능성이 있는지 떠올리게 한 뒤, 작업을 맡을 대상인 Claude Code를 정의하겠습니다.

## Slide 5 — Claude Code는 말을 듣고 도구를 쓰는 작업 파트너다

[약 2분]

Claude Code는 질문에 답만 하는 채팅창이 아니라 파일을 읽고, Plan을 만들고, 터미널 명령을 실행할 수 있는 작업 파트너입니다. 화면의 다섯 단계를 왼쪽부터 짚으며 “읽기와 확인 사이에 사람이 개입한다”는 구조를 설명합니다.

도구가 더 많은 일을 할수록 사람의 역할은 줄어드는 것이 아니라 더 선명해집니다.

## Slide 6 — 사람은 목적과 금지사항과 완료 기준을 정한다

[약 2분]

좋은 위임은 사람이 목적, 금지사항, 완료 기준을 먼저 정하는 데서 시작합니다. “비개발자 동료가 내 역할을 이해한다”, “비공개 자료를 넣지 않는다”, “모바일 화면과 URL을 확인한다”처럼 관찰 가능한 문장으로 바꿔 말합니다.

이 기준이 실제로 적용되는 물리적 범위, 즉 작업 폴더를 살펴보겠습니다.

## Slide 7 — 작업 폴더는 책상이고 파일과 터미널은 도구다

[약 2분]

작업 폴더를 책상에 비유합니다. 책상 위의 `index.html`, `styles.css`, `app.js`는 재료이고, 터미널은 현재 상태를 확인하는 도구입니다. 저장소 밖 파일을 건드리지 말라는 문장이 왜 필요한지 폴더 트리와 `git status`를 함께 보며 설명합니다.

이 도구들을 한 번에 쓰지 않고 작은 확인 고리로 묶는 방법이 다음 장입니다.

## Slide 8 — 안전한 작업은 확인부터 Diff까지 반복한다

[약 2분]

현재 상태 확인, Plan 검토, 제작 승인, 화면과 Diff 확인을 한 바퀴로 읽습니다. 한 번에 “다 만들어줘”라고 맡기면 잘못된 방향을 늦게 발견하지만, 이 고리를 반복하면 수정 범위가 작을 때 멈출 수 있습니다.

고리의 첫 설계 문서인 Plan을 조금 더 구체적으로 보겠습니다.

## Slide 9 — Plan은 만들기 전에 보는 작업 도면이다

[약 1분]

Plan은 멋진 설명문이 아니라 실행 전에 확인하는 도면입니다. 파일, 화면 구성, 검증 방법, 배포 전 확인이라는 네 항목이 있어야 “무엇을 어디까지 할지”를 승인할 수 있습니다. 하나라도 비어 있으면 제작 전에 보완한다고 말합니다.

Plan이 범위를 정한다면 권한 확인은 행동의 문을 지킵니다.

## Slide 10 — 권한 확인은 실수를 막는 문지기다

[약 1분]

읽기와 조사는 먼저 진행할 수 있지만 파일 수정은 범위를 확인하고, Commit·Push·삭제는 사람이 승인한 뒤 실행합니다. 권한 질문을 번거로운 중단이 아니라 사고가 나기 전에 판단할 기회로 해석하도록 안내합니다.

권한을 줬더라도 결과는 반드시 별도의 증거로 확인해야 합니다.

## Slide 11 — “완료했습니다”보다 화면과 Git 상태를 본다

[약 2분]

브라우저에서는 제목, 링크, 줄바꿈, 가로 넘침을 보고 Git에서는 변경 파일과 실제 Diff를 봅니다. 한쪽만 보면 화면은 예쁘지만 비밀값이 들어가거나, 코드는 맞지만 모바일이 깨지는 문제를 놓칠 수 있습니다.

이제 같은 일을 약한 요청과 좋은 요청으로 비교해 보겠습니다.

## Slide 12 — “포트폴리오 만들어줘”에는 판단 기준이 없다

[약 2분]

화면의 한 문장을 그대로 읽고, 누가 볼지·무엇을 제외할지·어떻게 확인할지가 빠져 있음을 표시합니다. 이런 요청에서는 작업 파트너가 색상, 구성, 공개 범위를 임의로 채우게 되므로 결과를 평가할 공통 기준도 사라집니다.

빠진 요소를 네 칸으로 채우면 요청이 어떻게 달라지는지 보겠습니다.

## Slide 13 — 좋은 요청은 대상·결과·금지·확인의 네 칸이다

[약 2분]

대상은 비개발자 동료, 결과는 1분 안의 역할 이해, 금지는 개인정보와 비공개 자료 제외, 확인은 링크·넘침·변경 파일입니다. 참가자에게 각 칸을 자신의 포트폴리오 상황으로 한 문장씩 바꿔 보게 합니다.

[PRACTICE] 30초 동안 네 칸 가운데 가장 먼저 적을 수 있는 한 칸을 메모하게 합니다.

이 네 칸이 하나의 자연스러운 요청 문장에 들어간 모습을 다음 장에서 읽겠습니다.

## Slide 14 — 같은 목표도 요청이 구체적이면 결과가 달라진다

[약 1분]

강한 요청을 대상, 결과, 금지, 확인 순서로 끊어 읽습니다. 마지막의 “내 확인 전에는 Commit이나 Push를 하지 마”가 제작 범위뿐 아니라 외부 행동의 경계까지 정한다는 점을 짚습니다.

좋은 요청을 보내기 전에, 입력할 내용 자체가 공개 가능한지 분류해야 합니다.

## Slide 15 — 공개할 내용과 공개하면 안 될 내용을 먼저 나눈다

[약 1분]

역할과 일반화한 업무 방식은 공개 가능하지만 고객명, 내부 수치, 화면 캡처, 비밀값은 공개 금지입니다. 애매하면 원문을 쓰지 않고 “운영 개선”, “반복 업무 자동화”처럼 식별할 수 없는 수준으로 일반화합니다.

재료를 정했다면 바로 제작하지 않고 Plan 승인 절차로 넘어갑니다.

## Slide 16 — 제작 승인은 Plan의 네 항목을 확인한 뒤에 한다

[약 2분]

만들 파일, 한 페이지 구성, 데스크톱·모바일 확인, GitHub 공개 전 사람 확인을 차례로 짚습니다. 참가자에게 “검증 방법이 빠진 Plan을 받았다면 무엇을 요청할지” 묻고, 아직 수정하지 말라는 조건을 유지한 채 보완하게 한다고 답합니다.

이제 결과를 안전하게 기록하고 되돌리는 Git의 역할로 연결하겠습니다.

## Slide 17 — Git은 되돌아갈 수 있는 저장 지점을 만든다

[약 2분]

Git은 개발자만 쓰는 복잡한 도구가 아니라 확인된 상태를 남기는 기록 장치입니다. 화면을 확인하고 Commit한 뒤 Push한다는 순서를 화살표대로 읽으며, 검토 전에는 저장 지점을 만들지 않는다고 설명합니다.

첫 번째 Git 명령인 `git status`부터 보겠습니다.

## Slide 18 — git status는 지금 바뀐 것을 알려준다

[약 1분]

`git status`는 현재 어떤 파일이 수정되거나 새로 생겼는지 보여 줍니다. 예시의 두 파일이 우리가 의도한 대상인지 확인하고, 예상 밖 파일이 나오면 다음 명령으로 넘어가지 않고 원인을 찾습니다.

파일 이름을 확인한 뒤에는 실제 변경 줄을 `git diff`로 읽습니다.

## Slide 19 — git diff는 실제로 바뀐 내용을 보여준다

[약 2분]

빼기 줄은 이전 내용, 더하기 줄은 새 내용이라는 기본만 설명합니다. 제목이 의도대로 바뀌었는지, 고객명이나 비밀값이 새 줄에 섞이지 않았는지를 사람이 읽어야 합니다. Diff가 길면 파일별로 나눠 검토합니다.

확인한 변경을 로컬 기록으로 묶는 단계가 Commit입니다.

## Slide 20 — Commit은 설명이 붙은 저장 지점이다

[약 1분]

Commit 메시지 `feat: add portfolio page`가 어떤 상태를 저장했는지 설명해 준다고 말합니다. Commit은 내 컴퓨터의 Git 기록이며 아직 GitHub나 공개 URL에 반영된 것은 아니라는 경계를 분명히 합니다.

이 기록을 외부 저장소로 보내는 Push를 이어서 보겠습니다.

## Slide 21 — Push는 확인한 저장 지점을 GitHub로 보낸다

[약 2분]

`git push origin main`은 확인한 Commit을 GitHub의 main 브랜치로 보내는 외부 행동입니다. Push 직전 사람이 마지막으로 Diff와 공개 가능 여부를 확인하고, Push 뒤에는 GitHub 화면에서 같은 Commit이 보이는지 대조합니다.

개념을 익혔으니 이제 강사 화면에서 전체 고리를 실제로 실행하겠습니다.

## Slide 22 — 시연은 빈 저장소와 공개 가능한 문장으로 시작한다

[약 2분]

강사 시연용 `portfolio` 저장소가 비어 있음을 보여 주고, 사용할 소개 문장은 공개 가능한 역할과 일반화한 사례뿐이라고 다시 선언합니다. 이 시연에서는 삭제·Commit·Push를 하지 않고 첫 버전 검토까지 간다고 범위를 알려 줍니다.

[DEMO] 시연 터미널과 브라우저를 나란히 띄우고, 작업 파일이 아직 없음을 보여 줍니다.

첫 명령으로 현재 폴더와 Git 기준 상태를 확인하겠습니다.

## Slide 23 — 첫 화면에서 폴더와 Git 기준 상태를 확인한다

[약 2분]

`pwd`, `ls -la`, `git status`를 차례로 실행해 현재 위치, 파일 목록, 깨끗한 기준 상태를 보여 줍니다. 이후 생기는 변경을 정확히 구분하려면 시작점을 먼저 눈으로 확인해야 한다고 설명합니다.

[DEMO] 경로가 `portfolio`로 끝나고 `working tree clean`이 보이는 지점에서 멈춥니다.

이제 파일을 만들지 말고 조사만 요청하는 첫 프롬프트를 보내겠습니다.

## Slide 24 — 첫 프롬프트는 수정이 아니라 조사를 요청한다

[약 2분]

오른쪽 프롬프트를 함께 읽고 “아직 파일을 수정하지 말고”와 “저장소 밖의 파일은 건드리지 말고”를 강조합니다. 첫 요청의 목표는 제작이 아니라 Claude Code와 참가자가 동일한 현재 상태를 보는 것입니다.

[DEMO] 프롬프트를 붙여 넣고 Claude Code가 파일 목록, Git 상태, Plan을 설명하는 데서 멈춥니다.

돌아온 Plan을 네 가지 검토 항목으로 분해하겠습니다.

## Slide 25 — Plan에서 파일·화면·검증·배포 전 확인을 읽는다

[약 2분]

파일은 세 개뿐인지, 화면은 한 페이지인지, 검증 폭은 1440×900과 390×844인지, 공개 전 Diff 확인이 있는지 순서대로 체크합니다. 범위를 벗어난 설치나 배포가 들어 있다면 이 자리에서 삭제하게 합니다.

[DEMO] Plan 응답에서 네 항목을 커서로 짚고 아직 파일이 생기지 않았음을 `git status`로 확인합니다.

모든 항목이 맞을 때만 제한된 제작 승인을 내리겠습니다.

## Slide 26 — 범위가 맞을 때만 제작을 승인한다

[약 2분]

승인 문장의 `index.html`, `styles.css`, `app.js`를 읽으며 허용 파일을 고정합니다. 로컬 화면과 모바일 폭, 변경 파일을 보여 달라고 요청하되 삭제·Commit·Push 금지는 그대로 둡니다.

[DEMO] 승인 프롬프트를 보내기 직전에 참가자에게 “이 문장이 허용하지 않은 행동은 무엇인가”를 묻습니다.

승인 뒤 만들어지는 세 파일의 역할을 확인하겠습니다.

## Slide 27 — 첫 버전은 HTML·CSS·JavaScript 세 파일로 만든다

[약 2분]

`index.html`은 구조와 내용, `styles.css`는 시각적 위계와 반응형 배치, `app.js`는 작은 상호작용을 맡습니다. 별도 빌드 도구 없이 브라우저에서 바로 열 수 있게 범위를 단순하게 유지한 이유를 설명합니다.

[DEMO] `ls` 결과의 세 파일과 Claude Code의 변경 요약이 일치하는지 대조합니다.

생성 사실이 아니라 실제 렌더링을 두 화면 크기에서 보겠습니다.

## Slide 28 — 로컬 화면과 모바일 폭을 직접 확인한다

[약 2분]

[약 2분]

`git push origin main`은 외부 저장소를 바꾸는 행동이므로 실행 직전에 대상 remote와 branch를 확인합니다. 성공 메시지 뒤 GitHub 저장소를 새로고침해 최신 Commit 메시지와 파일이 방금 검토한 것과 같은지 봅니다.

[PRACTICE] 인증 오류가 반복되면 명령을 재시도하지 말고 로그인 상태와 remote 주소를 강사에게 보여 줍니다.

Pages를 켜기 전에 공개 준비 체크리스트를 한 번 더 닫겠습니다.

## Slide 53 — Pages 설정 전에 배포 체크 완료

[약 3분]

공개 가능한 내용, GitHub의 확인된 Commit, 데스크톱과 390×844, 모든 링크, `git status`를 마지막으로 확인합니다. 하나라도 불확실하면 Pages 설정으로 넘어가지 않고 수정, Diff, Commit, Push, 재확인의 고리로 돌아갑니다.

[PRACTICE] 참가자가 다섯 항목을 직접 체크하고, 짝에게 공개 준비 완료의 근거를 한 문장으로 설명하게 합니다.

체크가 끝났다면 GitHub Pages가 무엇을 배포하는지 살펴보겠습니다.

## Slide 54 — Pages는 확인한 정적 파일을 공개 URL로 연결한다

[약 1분]

GitHub Pages는 로컬 파일이 아니라 GitHub에 Push된 Commit의 정적 파일을 공개합니다. 따라서 로컬 화면이 최신이어도 Push되지 않았다면 공개 사이트에는 반영되지 않는다는 경계를 강조합니다.

실제 설정 화면에서 배포 branch와 폴더를 지정하겠습니다.

## Slide 55 — 설정 경로는 main과 /(root)를 선택한다

[약 1분]

저장소의 Settings, Pages, Deploy from a branch 순서로 이동하고 branch는 `main`, 폴더는 `/(root)`를 선택합니다. Save 뒤에도 선택값이 그대로 남아 있는지 화면에서 직접 확인합니다.

[DEMO] 클릭 경로를 천천히 보여 주되 다른 저장소의 설정을 바꾸지 않도록 저장소 이름을 먼저 읽습니다.

설정 저장과 공개 완료 사이에는 배포 시간이 필요합니다.

## Slide 56 — 배포는 잠시 기다린 뒤 공개 URL을 직접 연다

[약 2분]

배포 진행 중에는 설정을 반복해서 바꾸지 않고 완료 상태가 될 때까지 기다립니다. GitHub가 표시한 URL을 복사해 새 탭에서 열고 제목, 카드, 링크를 확인한 뒤 390×844에서도 다시 봅니다.

[DEMO] 공개 URL을 새 탭으로 열고 새로고침한 다음 링크 하나와 모바일 폭을 실제로 확인합니다.

만약 404가 보인다면 추측 대신 정해진 순서로 원인을 좁힙니다.

## Slide 57 — 404가 나오면 Git 상태부터 순서대로 확인한다

[약 1분]

첫째 로컬 Git 상태와 최신 Commit, 둘째 GitHub에 Push된 파일, 셋째 Pages의 `main / (root)`, 넷째 `index.html` 위치와 URL을 확인합니다. 설정 버튼을 반복해서 누르기보다 앞 단계의 증거부터 대조한다고 설명합니다.

가장 흔한 첫 원인인 로컬과 GitHub의 동기화 차이를 보겠습니다.

## Slide 58 — Push하지 않은 변경은 GitHub Pages에 보이지 않는다

[약 2분]

`git status`와 `git diff`에 남은 변경은 아직 GitHub Pages의 재료가 아닙니다. 승인한 실제 파일만 Add하고 Commit한 뒤 Push하며, GitHub 최신 Commit과 로컬에서 읽은 내용이 같은지 확인합니다.

[DEMO] 로컬 변경과 GitHub 최신 Commit의 차이를 예시로 가리키고, 검토 없이 무조건 Add하지 않는다고 강조합니다.

동기화가 맞다면 시작 파일의 위치와 공개 주소 경로를 확인하겠습니다.

## Slide 59 — index.html 위치와 프로젝트 URL 경로를 확인한다

[약 2분]

`main` 브랜치의 `/(root)`에 `index.html`이 있는지 GitHub 파일 목록에서 확인합니다. 프로젝트 사이트 주소에는 보통 저장소 이름이 포함되므로 주소를 손으로 추측하지 말고 Pages 화면에서 복사합니다.

[DEMO] 계정 루트 URL과 프로젝트 URL의 차이를 주소창에서 비교해 보여 줍니다.

404보다 더 중요한 사고인 민감정보 노출 대응도 짚고 넘어가겠습니다.

## Slide 60 — 민감정보를 노출했다면 삭제만 하지 말고 자격 증명을 폐기한다

[약 1분]

공개 저장소에 토큰, 비밀번호, 키가 올라갔다면 파일에서 지우는 것만으로 충분하지 않습니다. 이미 복사됐다고 가정하고 즉시 비활성화하거나 회전한 뒤 새 자격 증명은 공개 파일이 아닌 안전한 비밀 관리 위치에 둡니다.

사고 대응 원칙까지 확인했으니 최종 완료 증거를 한 화면에 모으겠습니다.

## Slide 61 — 공개 URL·Git 동기화·모바일 화면이 완료 증거다

[약 2분]

공개 URL을 새 탭에서 열어 링크를 클릭하고, `git status`와 GitHub 최신 Commit을 대조하고, 390×844에서 넘침과 터치 대상을 확인합니다. 서로 독립적인 세 증거가 모두 있어야 오늘의 결과가 완성됩니다.

[PRACTICE] 각자 세 증거를 보여 줄 수 있는 화면을 준비하고 짝에게 하나씩 확인받습니다.

마지막으로 이 고리를 다음 업무에 어떻게 재사용할지 정리하겠습니다.

## Slide 62 — 다음 업무에서는 같은 루프를 더 작은 작업에 적용한다

[약 2분]

다음 업무에서는 먼저 작은 결과와 완료 기준을 정하고, Plan 범위를 승인하고, 화면과 Diff를 확인한 뒤 Commit·Push·공개 증거를 남깁니다. 포트폴리오가 아니라 문서 수정이나 간단한 자동화에도 같은 고리를 적용할 수 있습니다.

[PRACTICE] 참가자에게 다음 주 적용할 작은 업무 하나와 확인할 증거 하나를 메모하게 합니다.

오늘의 모든 단계를 한 문장으로 닫겠습니다.

## Slide 63 — 시키고 끝내지 말고, 결과를 직접 확인한다

[약 1분]

Plan을 읽고 제작을 승인하고, 화면과 Diff를 보고, 확인한 Commit을 Push한 뒤 공개 URL을 직접 엽니다. Claude Code가 작업을 빠르게 해도 결과를 완성하는 사람은 증거를 확인하는 우리라는 점을 다시 말합니다.

참가자에게 자신의 공개 URL과 세 가지 완료 증거를 끝까지 확인해 달라고 요청하며 워크숍을 마칩니다.
