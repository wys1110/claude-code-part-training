작업 공간

작업 폴더는 책상이고
파일과 터미널은 도구다

저장소 밖 파일은 건드리지 않는다고 명시합니다.

├─ styles.css
└─ app.js

$ git status

안전한 반복

안전한 작업은 확인부터
Diff까지 반복한다

  1. 현재 상태를 확인한다
  2. Plan과 범위를 읽는다
  3. 사람이 제작을 승인한다
  4. 화면과 git diff를 확인한다

약한 요청

“포트폴리오 만들어줘”에는
판단 기준이 없다

약한 요청: 포트폴리오 하나 만들어줘.

누구를 위한지 · 무엇을 빼야 하는지 · 무엇을 확인할지 알 수 없습니다.

좋은 요청의 네 칸

좋은 요청은 대상·결과·금지·확인의 네 칸이다

대상

비개발자 동료

결과

1분 안에 역할을 이해

금지

개인정보·비공개 자료 제외

확인

링크·넘침·변경 파일

같은 목표, 다른 결과

같은 목표도 요청이 구체적이면
결과가 달라진다

강한 요청: 비개발자 동료가 1분 안에 내 역할을 이해하도록 한 페이지로 만들어줘.
모바일에서도 읽기 쉽게 하고, 개인정보와 회사 비공개 자료는 넣지 마.
완료 전에 깨진 링크, 화면 넘침, 변경 파일을 확인해줘.
내 확인 전에는 Commit이나 Push를 하지 마.

나의 역할, 일반화한 업무 방식, 공개된 성과

공개 금지

제작 승인

제작 승인은 Plan의 네 항목을
확인한 뒤에 한다

  1. 만들거나 바꿀 파일
  2. 한 페이지의 화면 구성
  3. 데스크톱·모바일 확인 방법
  4. GitHub 공개 전에 사람이 확인할 내용
Git

Git은 되돌아갈 수 있는
저장 지점을 만든다

확인 → Commit → Push 순서로 공개 가능한 상태를 남깁니다.

Git 상태

git status는 지금 바뀐 것을 알려준다

$ git status
modified: index.html
modified: styles.css

의도한 파일만 바뀌었는지 확인합니다.

변경 내용

git diff는 실제로 바뀐 내용을 보여준다

$ git diff
- <h2>임시 제목</h2>
+ <h2>개인 업무 포트폴리오</h2>

예상한 내용만 추가·수정되었는지 봅니다.

Commit

Commit은 설명이 붙은
저장 지점이다

$ git commit -m "feat: add portfolio page"
확인한 변경만 기록

Push

Push는 확인한 저장 지점을
GitHub로 보낸다

$ git push origin main

사람 확인 후 공개 URL을 연다
강사 시연 · 시작

시연은 빈 저장소와
공개 가능한 문장으로 시작한다

개인정보·비공개 자료·비밀값 없이, 한 페이지 개인 업무 포트폴리오를 만듭니다.

portfolio/
└─ (아직 작업 파일 없음)

01 · 기준 상태

첫 화면에서 폴더와 Git
기준 상태를 확인한다

작업은 이 저장소 안에서만 합니다.

$ pwd
.../portfolio

$ ls -la
.
..
.git/

$ git status
working tree clean

02 · 조사 요청

첫 프롬프트는 수정이 아니라
조사를 요청한다

이 저장소에 개인 업무 포트폴리오를 만들고 싶어.

먼저 현재 파일과 Git 상태를 확인해줘.
아직 파일을 수정하지 말고 아래 내용을 Plan으로 보여줘.
- 만들거나 바꿀 파일
- 화면 구성
- 데스크톱과 모바일 확인 방법
- GitHub Pages 공개 전에 내가 확인할 내용

저장소 밖의 파일은 건드리지 말고, 삭제·Commit·Push는 내 확인 전에 하지 마.

03 · Plan 검토

Plan에서 파일·화면·검증·배포 전 확인을 읽는다

파일

index.html, styles.css, app.js만 새로 만듭니다.

화면과 검증

소개·업무 방식·연락처 한 페이지. 1440×900과 390×844에서 직접 확인합니다.

공개 전

Diff에 비공개 내용이 없는지 보고, GitHub Pages 설정과 URL은 사람이 확인합니다.

04 · 제작 승인

범위가 맞을 때만
제작을 승인한다

Plan이 맞아.
index.html, styles.css, app.js만 만들어줘.
완료 후 로컬 화면, 모바일 폭, 변경 파일을 보여줘.
삭제·Commit·Push는 아직 하지 마.

05 · 첫 버전

첫 버전은 HTML·CSS·JavaScript 세 파일로 만든다

$ ls
index.html
styles.css
app.js

index.html → 소개·업무 방식·연락처
styles.css → 색·간격·반응형 레이아웃
app.js → 메뉴 또는 링크 상호작용

06 · 화면 확인

로컬 화면과 모바일 폭을
직접 확인한다

데스크톱 · 1440×900

첫 화면의 위계가 읽히는가?
여백과 카드가 넘치지 않는가?
링크가 동작하는가?

모바일 · 390×844

가로 스크롤이 없는가?
글자와 버튼을 누를 수 있는가?
내용이 잘리지 않는가?

07 · 관찰 가능한 수정

“더 예쁘게”를 관찰 가능한
수정 요청으로 바꾼다

모호한 말

“첫 화면을 더 예쁘게 해줘.”

확인 가능한 말

“390×844에서 소개 문단과 첫 카드 사이 간격을 줄이고, 제목이 세 줄을 넘지 않게 해줘. 수정 뒤 두 폭에서 다시 보여줘.”

08 · 검토 완료

Diff와 변경 파일이 맞으면
시연 1차 완료다

$ git status --short
?? index.html
?? styles.css
?? app.js

$ git diff --check
공백 오류 없음 · 의도한 세 파일만 확인

이제 사람이 내용을 검토하고 Commit·Push·공개 여부를 결정합니다.
