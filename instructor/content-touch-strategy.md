# 세부 내용 터치 전략

<div class="badge-row"><span>목적: 깊이 조절</span><span>기준: 설명·시연·실습</span><span>대상: 강사·자료 작성자</span></div>

## 전략의 핵심

2시간 강의에서 모든 기능을 동일한 깊이로 다루면 참가자는 핵심 흐름을 기억하지 못합니다. 각 주제를 다음 세 단계로 나눕니다.

| 등급 | 다루는 방식 | 참가자 기대 수준 |
|---|---|---|
| **A — Core** | 설명 + 화면 시연 + 직접 실습 + 확인 질문 | 혼자 재현 가능 |
| **B — Working** | 개념 설명 + 짧은 시연 + 사용 시점 | 도움을 받아 적용 가능 |
| **C — Awareness** | 문제와 해결 방향만 소개 | 존재와 도입 기준 인지 |

## 전체 주제 깊이 지도

|---|:---:|---:|---|

| 보안·민감정보 | A | 5분 | 입력 가능 여부와 권한을 사람이 판단한다 |

## 주제별 터치 방식

# 1. VS Code

## 전달 목표

VS Code 기능을 많이 가르치는 것이 아니라 **Claude Code 작업을 검증할 수 있는 화면**으로 인식시킵니다.

## 화면에서 반드시 터치

1. 저장소 루트를 Explorer에서 가리킴
2. 통합 터미널의 현재 경로 표시
3. Source Control의 변경 파일 목록
4. 파일 클릭 후 diff
5. Claude Code 패널 또는 터미널 세션

## 말로만 언급

- 디버거
- Extension Marketplace 구조
- 설정 동기화
- Remote Development

## 참가자 행동

- 파일 3개 찾기
- `git status` 실행
- Source Control 열기
- 변경 파일을 클릭해 diff 읽기

## 오해 교정

**오해:** VS Code에서 프로젝트가 열렸으니 터미널도 같은 위치다.

**교정:** 터미널의 경로를 직접 확인하게 합니다.

**오해:** 초록색 테스트 결과면 변경은 안전하다.

**교정:** 일부 요구만 검증할 수 있으므로 diff를 직접 보게 합니다.

## 전환 문장

> 이제 VS Code가 현재 파일과 변경을 보여준다면, Git은 그 변경을 안전한 단위로 저장합니다.

---

# 2. Git·GitHub

## 전달 목표

복잡한 버전 관리 이론이 아니라 **작업 위치와 변경 상태를 잃지 않는 법**을 가르칩니다.

## 반드시 터치

- `git status`
- 현재 branch
- `git diff`
- 파일별 stage
- commit
- push 대상 branch
- PR base와 compare

## 의도적으로 생략

- rebase 상세
- cherry-pick
- submodule
- worktree
- merge conflict 실습
- 강제 push

## 시각화 방식

한 장에서 다음만 반복합니다.

```text
내 PC(Local)             GitHub(Remote)
작업 파일 → Commit  ──Push──> Branch → Pull Request
```

## 참가자 행동

- `practice/<github-id>` branch 생성
- branch 이름을 소리 내어 확인
- 변경 전에 `git status`
- commit 전에 stage 파일 확인

## 오해 교정

**오해:** Save와 Commit은 같다.

**교정:** Save는 파일 저장, Commit은 Git 변경 이력의 저장점이라고 구분합니다.

**오해:** Push하면 main에 바로 반영된다.

**교정:** 자신의 branch를 원격에 올리고 PR을 통해 리뷰한다고 설명합니다.

## 전환 문장

> Git이 안전한 작업 단위를 만든다면, Claude Code는 그 단위 안에서 분석·수정·검증을 빠르게 돕습니다.

---

# 3. Claude Code 개념

## 전달 목표

참가자가 Claude Code를 “답변 생성기”가 아니라 “도구를 사용하는 작업 에이전트”로 이해하게 합니다.

## 설명 프레임

모든 예시를 다음 루프로 설명합니다.

```text
이해 → 계획 → 실행 → 검증 → 리뷰
```

한 단계당 하나의 실제 프롬프트를 보여줍니다. 추상적인 AI 구조 설명은 2분 이내로 제한합니다.

## 반드시 터치

- 프로젝트 파일 탐색
- 명령 실행
- 파일 수정
- 테스트
- Git diff
- 권한 승인

## 오해 교정

**오해:** 자세한 프롬프트를 한 번 주면 완벽하게 끝난다.

**교정:** 작업은 반복 루프이며 중간 계획과 결과를 검토해야 한다고 설명합니다.

**오해:** Claude가 코드베이스 전체를 항상 완전히 이해한다.

**교정:** 필요한 파일을 탐색하지만 누락하거나 잘못 해석할 수 있으므로 근거와 테스트를 요구합니다.

## 질문

> Claude가 수행할 수 있는 것과 사람이 책임져야 하는 것은 각각 무엇인가요?

# 4. Plan Mode와 권한

## 전달 목표

Plan Mode를 단순 기능이 아니라 **변경 전 리뷰 게이트**로 이해시킵니다.

## 직접 보여줄 것

- 권한 모드 위치
- Plan Mode 전환
- 제안된 계획
- 계획의 파일·변경·테스트 항목
- 계획 수정 또는 거부

## 계획 검토 체크리스트

```text
[ ] 요구사항을 맞게 이해했는가?
[ ] 수정 파일이 과도하지 않은가?
[ ] 기존 동작에 미치는 영향은?
[ ] 테스트 방법이 구체적인가?
[ ] 관련 없는 리팩터링이 포함됐는가?
```

## 피할 방식

- 모든 작은 요청에 Plan Mode 강제
- 계획 내용을 읽지 않고 승인
- 계획과 다른 변경이 발생해도 계속 진행

## 전환 문장

> Plan Mode가 이번 작업의 계획을 관리한다면, CLAUDE.md는 프로젝트마다 반복되는 기본 규칙을 관리합니다.

---

# 5. CLAUDE.md

## 전달 목표

긴 프로젝트 설명서가 아니라 Claude Code가 매번 따라야 할 **짧은 운영 규칙**으로 설명합니다.

## 좋은 항목

- 실제 설치·테스트·빌드 명령
- 코드 스타일과 변경 금지 범위
- 완료 전 검증 명령
- 프로젝트의 중요한 구조적 결정

## 나쁜 항목

- 비밀번호·토큰
- 일회성 요청
- 긴 역사 설명
- 코드에서 바로 알 수 있는 내용
- 실행되지 않는 명령

## 시연

현재 저장소의 `CLAUDE.md`에서 Commands와 Rules만 보여줍니다. 새 파일을 처음부터 길게 작성하지 않습니다.

# 6. Skills·Subagents·MCP·Hooks

## 공통 전략

기능 정의보다 **어떤 반복 문제를 해결하는지**를 먼저 말합니다.

| 기능 | 문제 | 교육에서 보여줄 수준 |
|---|---|---|
| Skills | 같은 절차를 매번 설명 | 파일 위치와 사용 예 |
| Subagents | 큰 조사로 메인 맥락 소모 | 한 줄 요청 예 |
| MCP | 외부 시스템 데이터 필요 | 연결 사례와 권한 주의 |
| Hooks | 반드시 실행할 검사 필요 | formatter·lint 사례 |

## 시간 제한

전체 5분을 넘기지 않습니다. 참가자가 Core 흐름을 못 따라오면 이 구간을 2분 요약으로 줄입니다.

## 고급 기능을 깊게 안 다루는 이유

- 계정과 환경에 따라 설정이 다름
- 보안 검토가 필요함
- 초기 사용자는 기본 도구만으로도 대부분의 작업 가능
- 핵심 루프를 익힌 뒤 도입해야 효과를 판단할 수 있음

# 7. 통합 라이브 데모

## 전달 목표

도구별로 끊긴 지식을 하나의 실제 작업으로 연결합니다.

## 강사의 화면 규칙

- 한 번에 창 2개 이상 전환하지 않음
- Claude의 긴 설명은 핵심 네 줄만 읽음
- 터미널 출력은 실패 이름과 최종 결과만 확대
- diff에서 변경된 줄을 포인터로 직접 가리킴

## 중간 질문 네 번

1. branch 확인 후: “지금 main이 아닌가?”
2. 계획 후: “불필요한 파일이 포함됐는가?”
3. 테스트 후: “아직 무엇을 확인해야 하는가?”
4. commit 전: “어떤 파일이 stage됐는가?”

## 성공 기준

코드가 완벽히 수정되는 것보다 다음 흐름을 보여주면 성공입니다.

```text
상태 확인 → 증상 재현 → 근거 분석 → 계획 승인 → 수정 → 검증 → diff → commit
```

# 8. 참가자 실습

## 전략

실습 20분 동안 참가자가 막히지 않도록 **한 번에 한 단계만 공개**합니다.

### 1단계 — 탐색

```text
아직 수정하지 말고 구조와 테스트 목적을 설명해줘.
```

### 2단계 — 실패 재현

```text
테스트를 실행하고 증상과 근본 원인을 구분해줘.
```

### 3단계 — 계획

```text
최소 수정 계획, 영향 범위, 검증 방법을 제시해줘.
```

### 4단계 — 수정·검증

```text
합의한 범위만 수정하고 전체 테스트를 실행해줘.
```

### 5단계 — diff·commit

```bash
git diff
git status
```

## 강사가 바로 정답을 주지 않는 질문법

- 현재 경로를 확인해 볼까요?
- `git status`에는 무엇이 나오나요?
- 실패한 테스트 이름은 무엇인가요?
- Claude가 근거로 든 파일을 직접 열어봤나요?
- diff에서 요청하지 않은 파일이 있나요?

# 9. 보안과 팀 적용

## 공포 중심 설명 금지

“AI는 위험하다”로 끝내지 않고, 각 위험에 대응 행동을 연결합니다.

| 위험 | 행동 |
|---|---|
| 민감정보 입력 | 정책 확인 전 공개 예제만 사용 |
| 과도한 권한 | 읽기·Plan부터 시작 |
| 광범위한 수정 | branch와 최소 범위 명시 |
| 잘못된 결과 | 테스트·diff·사람 리뷰 |
| 외부 연결 | 데이터 범위·쓰기 권한 검토 |

## 팀 적용 메시지

```text
작은 개인 작업
→ 공통 CLAUDE.md
→ 반복되는 Skill
→ 필요한 Hook
→ 검토된 MCP
```

## 마무리 질문

> 자동화할 수 있는 가장 큰 작업이 아니라, 다음 주 안전하게 시험할 가장 작은 작업은 무엇인가요?

## 슬라이드 작성 규칙

- 한 장에 메시지 하나
- 설명 슬라이드 3장 뒤에는 시연 또는 질문
- 코드 블록은 8줄 이내
- 메뉴 설명은 스크린샷에 번호 표시
- 표는 최대 5행
- 고급 기능 슬라이드는 “정의”보다 “언제 쓰는가” 중심
- 발표자 노트에 예상 질문과 실패 대안을 기록
