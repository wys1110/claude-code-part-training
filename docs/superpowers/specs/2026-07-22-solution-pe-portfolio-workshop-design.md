# Solution PE팀 Claude Code 실무 워크숍 초안 설계

## 커뮤니케이션 목표

발표가 끝나면 Solution PE팀 Staff가 Claude Code를 기능 목록이 아니라 안전한 작업 파트너로 이해하고, 빈 GitHub 저장소에서 개인 업무 포트폴리오를 만든 뒤 검토·수정·공개 URL 확인까지 직접 완주할 수 있어야 한다.

발표자는 원용석이며, 개발 경험이 없는 참가자도 따라올 수 있는 쉬운 한국어를 사용한다.

## 결과물

- 기존 발표자료를 덮어쓰지 않는 별도 HTML 초안
- `make-slide`의 `minimal-dark` 테마를 적용한 단일 파일 프레젠테이션
- 전체 발표자 스크립트
- 브라우저에서 바로 열 수 있는 비교·검토용 URL
- 초안 검토 후 내용 강화 여부를 결정할 수 있는 명확한 섹션 구조

초안 경로는 다음으로 고정한다.

- `drafts/solution-pe-portfolio-workshop/index.html`
- `drafts/solution-pe-portfolio-workshop/script.md`

## 시간과 규모

총 교육 시간은 120분을 유지한다. 장수 제한은 두지 않되, 초안은 약 60~65장을 목표로 한다. 한 장에는 하나의 주장이나 하나의 참가자 행동만 둔다.

| 구간 | 시간 | 예상 장수 | 역할 |
| --- | ---: | ---: | --- |
| 오늘 만들 결과 | 5분 | 4장 | 완성된 포트폴리오와 공개 URL을 먼저 보여준다. |
| Claude Code 핵심 | 12분 | 7장 | 작업 폴더, 요청, Plan, 권한, 검증을 사례로 설명한다. |
| 좋은 요청과 안전 | 8분 | 5장 | 약한 요청을 강한 요청으로 바꾸고 금지사항을 정한다. |
| Git과 GitHub 핵심 | 8분 | 5장 | Status, Diff, Commit, Push를 결과물 이동 과정으로 설명한다. |
| 강사 전체 시연 | 17분 | 9장 | 빈 저장소부터 첫 버전 확인까지 전체 흐름을 보여준다. |
| 휴식과 환경 점검 | 10분 | 1장 | 로그인, 저장소, 터미널 준비 상태를 확인한다. |
| 참가자 단계별 실습 | 45분 | 22장 | 개인 업무 포트폴리오를 직접 제작하고 개선한다. |
| 공개와 실패 대응 | 10분 | 7장 | GitHub Pages 공개, 404, 미푸시, 경로 문제를 해결한다. |
| 완료 증거와 마무리 | 5분 | 3장 | 공개 URL과 Git 상태로 완료를 증명한다. |

## 학습 서사

자료는 기능 소개가 아니라 하나의 연속 사례로 진행한다.

1. 완성된 개인 업무 포트폴리오를 먼저 본다.
2. Claude Code에 무엇을 맡기고 무엇을 사람이 판단하는지 이해한다.
3. 모호한 요청이 왜 허술한 결과를 만드는지 실제 사례로 확인한다.
4. 현재 상태 확인과 Plan을 통해 안전하게 시작한다.
5. 강사 시연으로 전체 흐름을 한 번 본다.
6. 참가자가 같은 흐름을 Step 단위로 따라 한다.
7. 첫 결과를 직접 보고 구체적인 개선 요청을 작성한다.
8. Diff와 Git 상태를 확인한 뒤 GitHub에 기록한다.
9. 공개 URL을 열고 실패 상황을 해결한다.
10. 완료 메시지가 아니라 화면과 Git 상태를 증거로 남긴다.

## 개인 업무 포트폴리오 사례

실습 결과물은 한 페이지 정적 웹이다.

- 이름과 담당 업무
- 내가 해결하는 문제
- 핵심 역량 세 가지
- 대표 업무 또는 프로젝트
- 협업 방식
- 연락 또는 다음 행동
- 데스크톱과 모바일 대응

실제 회사 자료, 고객·임직원 정보, 비공개 수치, 토큰, 내부 URL은 사용하지 않는다. 실습에서는 공개 가능한 가상 문구와 참가자가 직접 정리한 일반 업무 설명만 사용한다.

## 12단계 실습

1. 완성 결과와 완료 기준 확인
2. 포트폴리오에 넣을 공개 가능 정보 준비
3. GitHub Public 저장소 생성
4. 저장소 Clone과 작업 폴더 확인
5. 현재 파일과 `git status` 조사 요청
6. 수정 파일·화면 구성·검증 방법을 Plan으로 받기
7. `index.html`, `styles.css`, `app.js` 첫 버전 제작
8. 로컬 브라우저와 모바일 폭에서 직접 확인
9. 색상·여백·정보 순서를 포함한 구체적인 디자인 개선 요청
10. 변경 파일, 삭제, 민감정보, 깨진 링크 검토
11. `git status → git diff → git add → git commit → git push`
12. GitHub Pages 활성화와 공개 URL 검증

각 실습 Step은 필요에 따라 1~3장으로 나누고 다음 순서를 유지한다.

1. 이번 단계의 결과
2. 강사가 보여주는 실제 사례
3. 참가자가 복사할 프롬프트 또는 명령
4. 정상이라면 보이는 결과
5. 사람이 반드시 확인할 사항
6. 안 될 때 가장 먼저 확인할 항목
7. 완료 체크

## 내용을 강하게 만드는 규칙

- 모든 핵심 개념은 추상 설명 직후 개인 업무 포트폴리오 사례로 연결한다.
- Claude Code의 기능명을 나열하지 않고 실제 요청과 파일 변화로 설명한다.
- 약한 프롬프트와 강한 프롬프트를 나란히 보여준다.
- Plan, Diff, `git status`, Commit 결과는 실제 출력 형태로 보여준다.
- 실수 사례를 숨기지 않고 404, 미푸시, 잘못된 폴더, 민감정보 포함 상황을 다룬다.
- 각 사례는 무엇이 달라졌는지와 왜 확인해야 하는지까지 설명한다.
- 화면의 짧은 문구는 결론을 전달하고, 자세한 설명·전환·질문은 발표자 노트에 둔다.

## 시각 디자인

- `make-slide`의 `minimal-dark`를 시각 기준으로 사용한다.
- 배경은 검정, 기본 글자는 흰색, 핵심 행동과 완료 증거는 파란색으로 강조한다.
- 제목·섹션 구분·마무리는 중앙 정렬 전체 화면을 사용한다.
- 설명과 결과 화면, 약한 요청과 강한 요청, 명령과 예상 결과는 Split 레이아웃을 사용한다.
- 긴 명령·프롬프트·Diff는 Wide 레이아웃을 사용한다.
- 반복되는 카드 격자 대신 큰 제목, 실제 터미널, 브라우저 프레임, 체크리스트를 중심으로 구성한다.
- 초안에서는 외부 이미지를 사용하지 않는다. CSS 기반 터미널·브라우저 표현으로 내용과 구조를 먼저 검증한다.
- 초안 검토 후 실제 Claude Code, GitHub, GitHub Pages 화면 캡처가 학습에 필요한 장면만 선별해 추가한다.

## 발표자 노트

모든 슬라이드에 원용석 발표 기준의 자연스러운 한국어 노트를 넣는다.

- 해당 장에서 강조할 한 문장
- 쉬운 비유 또는 사례 설명
- 참가자에게 던질 질문
- 시연 또는 실습 시작·중지 지점
- 정상 결과와 실패 시 확인 순서
- 다음 장으로 이어지는 전환 문장
- 섹션별 시간 안내

`S` 키로 별도 발표자 노트 창을 열 수 있어야 하며, 노트는 화면 본문에 노출하지 않는다.

## 기술 구조

- Vanilla HTML, CSS, JavaScript만 사용한다.
- 발표자료는 CSS와 JavaScript가 모두 포함된 단일 `index.html`로 만든다.
- 별도 빌드나 패키지 설치 없이 브라우저에서 실행되어야 한다.
- 키보드 좌우 이동, Space 이동, 터치 스와이프, 전체 화면, 진행 표시, 페이지 번호, 인쇄, 발표자 노트를 제공한다.
- 초안은 기존 루트 발표자료와 분리하며, 사용자 승인 전 루트 `index.html`을 교체하지 않는다.

## 검증

- 슬라이드 수가 목표 범위인 55~70장인지 확인한다.
- 발표자 노트에 기록한 전체 시간이 120분인지 확인한다.
- 모든 슬라이드에 제목 또는 명확한 참가자 행동이 있는지 확인한다.
- 12단계 실습과 두 개 이상의 복사 가능한 프롬프트가 포함되는지 확인한다.
- 금지된 민감정보와 부정확한 권한 모드명이 없는지 검사한다.
- 모든 슬라이드를 1440×900에서 렌더해 겹침·잘림·과도한 줄바꿈을 확인한다.
- 주요 실습 장면을 390×844에서 확인한다.
- 마지막 장, 좌우 이동, 전체 화면, 발표자 노트, 인쇄가 동작하는지 확인한다.
- 브라우저 콘솔 오류가 없는지 확인한다.

## 초안 완료 기준

- Solution PE팀 Staff와 발표자 원용석이 표지와 발표자 노트에 반영된다.
- 개인 업무 포트폴리오 사례가 발표 처음부터 공개 URL 확인까지 이어진다.
- 이론보다 시연과 참가자 행동이 더 많은 시간을 차지한다.
- 실습이 한 번에 크게 뛰지 않고 Step 단위로 따라갈 수 있다.
- 참가자가 복사할 프롬프트·명령과 정상 결과·확인사항을 함께 볼 수 있다.
- Minimal Dark 시각 체계가 모든 장에서 일관된다.
- 기존 발표자료는 보존되고 별도 초안 URL에서 검토할 수 있다.
