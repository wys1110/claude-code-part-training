# Claude Code 실무 워크숍 발표자료 재설계

## 커뮤니케이션 목표

발표가 끝나면 Solution PE팀 Staff가 Claude Code를 어려운 개발 도구가 아니라 "요청을 받고 파일과 명령을 다루는 작업 파트너"로 이해하고, 안전한 순서에 따라 정적 웹을 만든 뒤 GitHub Pages 공개 URL까지 직접 완성할 수 있어야 한다.

## 대상과 설명 원칙

- 대상은 Claude Code, Git, 웹 개발 경험이 고르지 않은 사내 Staff다.
- 전문가가 초등학생에게 설명하듯 쉬운 한국어를 사용한다.
- 어려운 기능명은 먼저 일상적인 비유로 설명한 뒤 정확한 이름을 붙인다.
- 슬라이드 한 장에는 핵심 주장이나 참가자 행동을 하나만 둔다.
- 영어는 `Claude Code`, `GitHub`, `Pages`, `Plan`, `Test`, `Diff`, 명령어처럼 제품명과 기술 표기에만 쓴다.
- 화면에 긴 설명을 넣지 않는다. 자세한 설명과 전환 문장은 발표자 노트에 둔다.
- 참가자가 복사해야 하는 프롬프트와 명령은 그대로 실행 가능한 형태로 제공한다.

## 구성 방식

기존 120분 틀은 유지하되 44장을 약 28장으로 줄인다. 줄인 시간은 설명을 늘리는 데 쓰지 않고 강사 시연과 참가자 실습에 배정한다.

| 구간 | 시간 | 목적 |
| --- | ---: | --- |
| 시작 | 5분 | 오늘 만들 결과와 안전 규칙을 확인한다. |
| Claude Code 핵심 | 20분 | 작업 방식, 좋은 요청, Plan, 권한, 검증을 이해한다. |
| Git과 GitHub 핵심 | 15분 | 저장소, Branch, Commit, Push를 결과물 이동 과정으로 이해한다. |
| 강사 시연 | 20분 | 웹 제작부터 공개 URL 확인까지 전체 흐름을 먼저 본다. |
| 휴식과 환경 점검 | 10분 | 로그인, 저장소, 터미널 상태를 확인한다. |
| 참가자 실습 | 40분 | 각자 정적 웹을 만들고 GitHub Pages로 배포한다. |
| 실패 대응과 정리 | 10분 | 404, 미푸시, 경로, 민감정보 문제를 해결하고 원칙을 복습한다. |

## 이야기 흐름

1. 오늘의 결과: "내가 설명한 웹이 인터넷 주소로 열린다."
2. Claude Code의 역할: 말을 알아듣고 도구를 쓰는 작업 파트너다.
3. 사람의 역할: 목적, 금지사항, 완료 기준을 정하고 결과를 확인한다.
4. 안전한 반복: 확인 → 계획 → 만들기 → 테스트 → 차이 확인 → 기록.
5. Git과 GitHub: 내 컴퓨터의 작업을 안전하게 기록하고 원격 저장소에 올린다.
6. 강사 시연: 전체 과정을 실수와 확인 지점까지 포함해 완주한다.
7. 참가자 실습: 같은 프롬프트 구조로 각자의 웹을 만들고 배포한다.
8. 마무리: 공개 URL과 Git 상태라는 증거로 완료를 판단한다.

## 슬라이드 구성안

### 시작 — 3장

1. Claude Code로 웹을 만들고 인터넷에 공개하기
2. 오늘 끝나면 손에 남는 것: GitHub 저장소, Commit, 공개 URL
3. 먼저 지킬 세 가지: 공개 가능한 내용만, 삭제·Push 전 확인, 완료 메시지보다 결과 확인

### Claude Code 핵심 — 7장

4. Claude Code는 "말을 듣고 도구를 쓰는 작업 파트너"
5. 작업 폴더는 책상, 파일은 재료, 터미널은 도구함
6. 좋은 요청의 네 칸: 대상, 원하는 결과, 하지 말 것, 확인 방법
7. `plan`은 만들기 전 도면 보기
8. 기본 권한은 문을 열기 전 물어보는 안전장치
9. `CLAUDE.md`는 매번 다시 말하지 않을 팀 규칙 메모
10. 안전한 작업 루프: 확인 → 계획 → 수정 → Test → Diff

### Git과 GitHub 핵심 — 5장

11. Git은 저장 지점, GitHub는 원격 보관함
12. Branch는 원본을 지키는 연습장
13. Commit은 설명이 붙은 저장 지점
14. Push는 저장 지점을 GitHub로 보내기
15. Pages는 저장소의 정적 파일을 웹 주소로 공개하기

### 강사 시연 — 5장

16. 시연의 완성 그림과 준비 상태
17. 통합 프롬프트 읽기: Claude가 무엇을 하고 무엇을 하지 않는지 확인
18. Plan과 변경 파일 확인
19. 로컬 화면, Test, Diff 확인
20. Commit·Push·Pages·공개 URL 확인

### 휴식 — 1장

21. 10분 휴식과 실습 준비 체크

### 참가자 실습 — 5장

22. 빈 Public 저장소 만들기와 URL 복사
23. 저장소 Clone과 기준 상태 확인
24. 통합 프롬프트 실행
25. 결과 검토 후 Commit·Push
26. Pages 설정과 공개 URL 검증

### 실패 대응과 종료 — 3장

27. 안 될 때 확인 순서: Git 상태 → Push → Pages 상태 → `index.html` → URL
28. 공개하면 안 되는 것과 이미 노출했을 때의 대응
29. 기억할 한 줄: "시키고 끝내지 말고, 결과를 직접 확인한다."

구현 중 내용 밀도에 따라 27~30장 범위에서 조정할 수 있지만, 새로운 기능 소개를 추가해 장수를 늘리지는 않는다.

## 삭제와 통합

- Explorer, Terminal, Source Control, Diff, Command Palette를 각각 설명하던 장은 "작업 폴더와 도구" 및 "안전한 작업 루프"로 통합한다.
- Repository 생성, URL 복사, Claude 요청, 내부 작업, Pages API, 기능 확인으로 나뉜 여섯 장은 강사 시연과 참가자 실습 흐름으로 다시 묶는다.
- `Agent Loop`, 책임 분리, 기본 안전 원칙의 반복 문장은 한 개의 작업 루프로 통합한다.
- 권한 모드 장은 최신 공식 명칭인 `default`와 `plan` 중심으로 줄인다. `auto`, `dontAsk`, `bypassPermissions`는 초급 실습의 핵심이 아니므로 본문에서 제외하고 필요하면 발표자 노트에서만 주의사항으로 언급한다.
- Skills, Subagents, Plugins, MCP, Hooks, Agent Teams 지도는 이번 실습 결과에 필요하지 않으므로 삭제한다.
- Live Demo와 Participant Lab의 반복 체크리스트는 단계별 한 번씩만 보여준다.
- 기능 목록형 Q&A 장 대신 참가자의 공개 URL과 다음 업무 적용 행동으로 닫는다.

## 통합 실습 프롬프트 설계

프롬프트는 참가자가 저장소 루트에서 그대로 실행할 수 있도록 작성한다. 한 번에 모든 권한을 맡기지 않고 두 단계로 사용한다.

### 1단계: 계획

```text
이 저장소에 초보자도 이해할 수 있는 한 페이지 소개 웹을 만들고 싶어.

먼저 현재 파일과 Git 상태를 확인해줘.
아직 파일을 수정하지 말고 아래 내용을 계획으로 보여줘.
- 만들거나 바꿀 파일
- 화면 구성
- 확인 방법
- GitHub Pages 배포 전에 내가 확인할 내용

조건:
- HTML, CSS, JavaScript만 사용해.
- 별도 설치나 빌드 과정은 만들지 마.
- 모바일과 컴퓨터 화면에서 모두 읽기 쉽게 해.
- 저장소 밖의 파일은 건드리지 마.
- 기존 파일을 삭제하거나 force push하지 마.
- 개인정보, 회사 자료, 토큰, 비공개 주소를 넣지 마.
```

### 2단계: 제작과 검증

```text
방금 합의한 계획대로만 만들어줘.

웹 내용:
- 제목: [내 웹 제목]
- 한 줄 설명: [내가 소개할 내용]
- 핵심 항목 세 개: [항목 1], [항목 2], [항목 3]

완료 전에 다음을 확인해줘.
1. index.html이 저장소 루트에 있는지
2. 깨진 링크나 화면 넘침이 없는지
3. 모바일에서도 읽을 수 있는지
4. 변경 파일이 요청 범위와 맞는지

확인이 끝나면 실행한 검증과 git diff 요약을 알려줘.
내 확인 전에는 commit이나 push하지 마.
```

검토가 끝난 뒤 참가자가 별도로 Commit과 Push를 요청한다. Pages 활성화는 GitHub의 `Settings → Pages → Deploy from a branch → main → /(root)`를 기본 경로로 사용한다. `gh` 인증과 저장소 권한이 준비된 경우에만 Claude Code가 API 설정을 보조할 수 있다.

## 시각 디자인

- 현재 어두운 배경과 주황·파랑 강조색은 유지해 기존 정체성을 보존한다.
- 상단의 고정 섹션명과 슬라이드 내부 kicker 중 하나만 보여 중복을 없앤다.
- 제목은 결론형 문장으로 쓰고 한 줄을 원칙으로 한다.
- 카드 격자는 꼭 비교가 필요한 경우에만 사용한다.
- 과정은 큰 숫자, 짧은 동사, 넓은 간격으로 표현한다.
- 코드나 프롬프트가 긴 장은 핵심 구문만 화면에 보여주고 전체 복사본은 별도 복사 영역이나 발표자 노트에 둔다.
- 하단 조작부와 본문 사이에 충분한 안전 여백을 둔다.
- 장식용 아이콘과 의미 없는 영문 라벨을 추가하지 않는다.
- 1440×900 데스크톱과 390px 모바일 폭에서 제목 줄바꿈, 본문 넘침, 조작부 겹침을 검증한다.

## 완료 기준

- 발표자료는 27~30장, 총 120분으로 계산된다.
- 모든 슬라이드가 한 가지 주장 또는 한 가지 행동에 집중한다.
- 초급 참가자가 복사 가능한 계획 프롬프트와 제작 프롬프트가 포함된다.
- 실습은 정적 HTML/CSS/JavaScript, Git Commit·Push, GitHub Pages 공개 URL 확인까지 이어진다.
- 최신 공식 용어를 사용하며 `Manual` 같은 부정확한 모드명은 남지 않는다.
- 대표 화면뿐 아니라 모든 슬라이드를 렌더해 겹침, 잘림, 과도한 줄바꿈을 확인한다.
- 키보드 이동, 개요, 노트, 전체 화면, 인쇄 기능이 유지된다.
- 공개 저장소에 넣으면 안 되는 정보와 실패 시 확인 순서가 포함된다.

## 범위 밖

- React, 프레임워크, 패키지 설치, 빌드 파이프라인
- GitHub Actions를 직접 작성하는 배포 실습
- Skills, Subagents, MCP, Hooks, Agent Teams 상세 교육
- 회사 코드, 사내 문서, 운영 로그, 고객·임직원 정보 사용
- 자동 force push, 기존 파일 일괄 삭제, 확인 없는 직접 `main` Push
