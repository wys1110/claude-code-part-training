# Claude Code 5개 스킬 통합 설계

## 목표

Solution PE팀 Staff가 Claude Code의 기본 작업 흐름을 이해한 뒤, 좋은 스킬이 설계·계획·디자인·디버깅·검증 절차를 보완해 프롬프트를 짧게 만들 수 있음을 같은 개인 업무 포트폴리오 사례로 체감하게 한다.

## 핵심 메시지

- 좋은 스킬은 긴 프롬프트를 대신하는 반복 가능한 업무 절차다.
- Claude Code가 상황을 감지해 적절한 스킬을 사용할 수 있지만, 처음 연습할 때는 스킬 이름을 직접 불러 차이를 관찰한다.
- 스킬을 사용해도 사람은 원하는 결과, 대상, 공개 금지 정보, 승인 경계, 완료 기준을 분명히 말해야 한다.

## 출처

사용자가 제공한 `docs/claude-code-beginner-skills-5.md`를 5개 스킬 설명과 예시 프롬프트의 기준으로 사용한다. 설치 명령처럼 변할 수 있는 내용은 구현 시 공식 Claude Code 문서와 원본 저장소에서 다시 확인한다. 인기 수치는 핵심 교육 흐름에 넣지 않는다.

## 발표 구조

기존 63장에 10장짜리 스킬 챕터를 추가해 73장으로 확장한다. 전체 시간은 120분을 유지한다.

1. 기존 슬라이드 1–11: Claude Code 기본 작업 흐름, 11분
2. 신규 슬라이드 12–21: 5개 스킬과 사용 전후 비교, 16분
3. 기존 요청·Git 기본기: 10분
4. 기존 강사 시연: 13분
5. 휴식·환경 확인: 10분
6. 기존 12단계 참가자 실습: 45분
7. Pages 공개·문제 해결·마무리: 15분

합계는 120분이다. 기존 2분짜리 개념 설명 중 반복되는 장을 1분으로 줄이고, 45분 실습과 10분 휴식은 유지한다.

## 신규 스킬 챕터

| 신규 ID | 분 | 핵심 제목 | 역할 |
| ---: | ---: | --- | --- |
| 12 | 1 | 스킬은 반복 가능한 업무 절차를 불러온다 | 스킬 개념 |
| 13 | 2 | 스킬이 없으면 절차를 긴 프롬프트로 직접 적는다 | 사용 전후 비교 |
| 14 | 1 | 처음에는 이름을 부르고 익숙해지면 상황에 맡긴다 | 스킬 센싱 원칙 |
| 15 | 2 | brainstorming은 만들기 전에 목적과 범위를 맞춘다 | 요구사항 정리 |
| 16 | 2 | writing-plans는 합의한 일을 실행 순서로 바꾼다 | 계획 |
| 17 | 2 | frontend-design은 예쁨보다 전달 목적을 설계한다 | 화면 개선 |
| 18 | 2 | systematic-debugging은 수정 전에 원인을 증명한다 | 오류 해결 |
| 19 | 2 | verification-before-completion은 완료를 증거로 바꾼다 | 완료 검증 |
| 20 | 1 | 입문자는 두 번의 설치로 다섯 스킬을 준비한다 | 설치 안내 |

각 스킬 장은 같은 포트폴리오 업무를 기준으로 `스킬 없음`과 `스킬 사용`을 비교한다. 단순 기능 목록 대신 행동 변화, 사람이 확인할 항목, 복사 가능한 짧은 요청을 보여준다.

## 기존 시연·실습 연결

- Plan 요청 장에는 `brainstorming`과 `writing-plans` 배지를 표시한다.
- 디자인 개선 장에는 `frontend-design` 배지를 표시한다.
- Pages 404 문제 해결 장에는 `systematic-debugging` 배지를 표시한다.
- 최종 공개 URL·모바일·Git 증거 확인 장에는 `verification-before-completion` 배지를 표시한다.
- 배지는 설명을 반복하지 않고 “지금 어떤 절차가 작동하는가”를 연결하는 표지로만 사용한다.

## 시각 설계

- 전체 덱의 Minimal Dark 기반은 유지한다.
- 신규 12–21번은 Neon Terminal 변형을 사용한다: 어두운 배경, 초록·청록 포인트, 터미널 커서/프롬프트를 연상시키는 장식.
- 복사 가능한 실제 명령에는 실행을 방해하는 `$` 프롬프트 문자를 넣지 않는다.
- 스킬별 색상을 과도하게 늘리지 않고 초록은 절차, 청록은 사람의 판단, 빨강은 위험 또는 실패 비교에만 사용한다.
- 1440×900에서 모든 내용을 한 화면에 읽을 수 있게 하고, 모바일 대표 화면에서는 가로 넘침이 없어야 한다.

## 발표자 스크립트와 노트

- `script.md`는 73개 섹션으로 확장하고 시간 합계 120분을 유지한다.
- 신규 장에는 같은 포트폴리오 사례, 비교 설명, 질문, DEMO/PRACTICE 행동, 다음 장 전환을 작성한다.
- HTML 내부 전체 발표자 노트 JSON도 73개로 갱신하고 `script.md`와 정확히 일치시킨다.

## 검증 기준

- 73개 슬라이드가 1–73 순서이며 시간 합계가 정확히 120분이다.
- 다섯 스킬 이름, 역할, 짧은 요청, 사람의 확인 책임이 모두 보인다.
- 사용 전후 비교가 최소 한 장의 통합 비교와 다섯 개 스킬별 비교에 존재한다.
- 설치 명령은 공식 출처로 현재성을 확인한다.
- 73개 발표자 스크립트와 HTML 내장 노트가 정확히 일치한다.
- 데스크톱 73장, 신규/고위험 모바일 대표 장, 73페이지 인쇄물을 렌더링해 개별 확인한다.
- 기존 12단계 실습의 실행 가능성과 Git·Pages 안전 절차를 보존한다.
- 루트 덱은 변경하지 않는다.
